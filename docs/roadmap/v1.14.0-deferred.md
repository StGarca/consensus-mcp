# v1.14.0 - deferred from v1.13.0 (dispatch reliability + visibility)

Per iter-0007 codex Q1 verdict (sealed packet `39f8b7a8be6e4d1998bfc4b43e00d6559bf1645a2cadb414ef7a486193c90c1b`), v1.13.0 shipped F1 (3-way root split) and F2 (spec template in wheel) only. Three findings about dispatch reliability and visibility were scoped to v1.14.0:

## F3 - Dispatcher auto-discovers `review-packet.yaml`

**Current behavior** (`_dispatch_codex.py:1746-1762`): `touched_files_contents` is loaded into the prompt only when `--review-target` points at a `.yaml` file. If the operator passes a raw `.md`, `.diff`, or `.patch` (the common-sense usage), the prompt's `{touched_files_contents_block}` ends up empty - codex either hallucinates from goal_packet context or refuses to verdict. iter-0005/0006-1/0006-3 demonstrated both failure modes.

**Proposed fix**: when `--review-target` is non-yaml or unspecified, the dispatcher auto-discovers `<iteration_dir>/review-packet.yaml`. If found, loads it for the touched-files-contents block and emits a `dispatch_review_packet_resolved` event with the path used. If neither the explicit yaml NOR the conventional path resolves, emit `dispatch_review_packet_missing` at WARNING level before sending the truncated prompt - so an operator looking at the dispatch log sees "warning: codex received prompt without embedded touched files" rather than silently getting a hallucinated verdict.

**Touches**: `_dispatch_codex.py` main() resolution path (~line 1746), audit-event emitter, test fixtures.

## F4 - Watchdog cold-start grace period

**Current behavior** (`_dispatch_codex.py:613`): `stall_silence_seconds: float = 45.0` applies uniformly from process launch. For bigger prompts, codex's pre-first-token LLM warmup exceeds 45s. iter-0006-2 was killed at 45s with `last_streamed_line_seq: null` - codex emitted zero bytes; it was still loading the model, not stalled.

**Proposed fix**: split into two thresholds -
- `pre_first_byte_silence_seconds: float = 180.0` - applies from process start until first stdout line
- `post_first_byte_silence_seconds: float = 45.0` - applies after first stdout line (current behavior renamed)

Operator overrides via env vars `CONSENSUS_MCP_PRE_FIRST_BYTE_SILENCE_SECONDS` and `CONSENSUS_MCP_POST_FIRST_BYTE_SILENCE_SECONDS`, plus CLI flags. The wall-time hard kill (`timeout_seconds + post_first_byte_silence_seconds`) stays.

**Touches**: `_dispatch_codex._invoke_codex` signature + watchdog loop (`_dispatch_codex.py:780-840`), `main()` arg parser, tests.

## F5 - Visibility: surface "loading" vs "stalled" state

**Current behavior**: `dispatch_heartbeat` events carry `last_streamed_line_seq: null` when codex hasn't produced output yet, but the event payload doesn't say what that means in human terms. Operators reading `dispatch-log.jsonl` can't distinguish "codex is loading, no output yet" from "codex was outputting, then went quiet."

**Proposed fix**: add a `status` field to dispatch_heartbeat events:
- `"loading"` - `last_streamed_line_seq is None`
- `"streaming"` - last line received within the last (post_first_byte_silence_seconds / 2) seconds
- `"stalled"` - last line older than that

Plus update `_visibility_tui.py` to render the status as a colored badge so an operator running the TUI in another terminal sees codex's state at a glance.

**Touches**: `_dispatch_codex.py` watchdog event emitter, `_visibility_tui.py` render path, tests.

## Sequencing

Suggested order: F3 -> F4 -> F5. F3 prevents the most common silent-failure mode (verdict on hallucinated premise). F4 unblocks bigger prompts. F5 is the operator-facing finish.
