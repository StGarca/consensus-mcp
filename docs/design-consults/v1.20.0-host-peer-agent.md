# v1.20.0 — Host-family specialist agents (converged design)

**Status:** converged via 4-way Workflow A consult (claude + codex + gemini +
kimi), iteration `host-family-agents-design-2026-05-22`. Implementation spec.
Workflow B review skipped per operator.

## What
1. A same-family **blind SWE-reviewer** contributor — `kind: host_peer` — that
   runs the host's own AI family (e.g. Claude when Claude hosts) as a *fresh-
   context, adversarial* code reviewer. It is **supplementary** and **excluded
   from the cross-family closure invariant** — it augments cross-family review,
   never replaces it.
2. A distinct **orchestrator role** (the existing in-process host): neutral
   scoping, synthesis, gate-enforcement, anti-anchoring — separated from the
   reviewer so the integrator never blind-reviews its own synthesis.

## Why
Same-model review shares blind spots, and same-family agreement is the shared-
prior trap (low information). But a *fresh-context* same-family reviewer with an
adversarial prompt is a cheap "second independent sample" that catches careless
slips, spec drift, and missed edge cases the host's anchored context overlooked —
without needing a CLI (the host has no `claude` shell binary).

## Decision (option (b), unanimous)
- **`KIND_HOST_PEER='host_peer'`** profile kind (beside `cli_reviewer` + reserved
  `host`); requires `family` + `role`.
- **`HostPeerAdapter(ContributorAdapter)`**: `dispatch()` calls a host review
  callback with the phase `DispatchPacket` only (structural blindness), seals a
  normal `SealedArtifact`, tags `family == host`, `role=swe_reviewer`,
  `gate_eligible=false`, `weight=supplementary`. Never shells out.
- **Dedicated callback** (separate from the orchestrator's
  `claude_artifact_callback`) so the reviewer seam is explicitly a fresh-context
  blind reviewer, never the orchestrator's own voice/context. Build only when the
  callback is wired; otherwise host_peer is gracefully absent.
- **Gate exclusion — two layers:** (1) explicit data — the closure check ignores
  any artifact with `gate_eligible==false` when seeking a different-family signer
  (so host_peer can NEVER close, even when its family differs from the mutator —
  the host is not independent of its own orchestration); (2) automatic — host_peer
  `family==host` already fails the existing cross-family check when host is the
  mutator. **Load-bearing regression test:** `[host + host_peer]` cannot close a
  host-authored change, and a claude host_peer cannot satisfy cross-family signoff
  for a codex-authored change — a genuinely external family is always required.
- **Role split:** distinct prompts — `host_peer_review_template.md` (adversarial:
  correctness, spec-conformance, edge cases) and `orchestrator_role.md` (neutral,
  anti-anchoring, synthesize-after-reveal). Enforced by separate prompts +
  contexts + the adapter kind. The dormant `can_review`/`can_converge` flags are a
  deferred nice-to-have (wiring them risks regressing existing dispatch).
- **Provenance:** `adapter=host_peer, family=<host>, role=swe_reviewer,
  weight=supplementary, gate_eligible=false, fresh_context_attestation`,
  prompt/output/goal hashes. Surfaced same-family so agreement isn't laundered as
  independent cross-family confirmation.
- **Fresh-context = host-runtime CONTRACT** (consensus-mcp cannot prove memory
  isolation): recorded via `independence_attestation{method:host_peer_callback,
  fresh_context:true, no_peer_review_visible_at_dispatch:true}`. Documented as a
  contract, not a mechanical guarantee. (All four reviewers flagged this honestly.)
- **Config:** `contributors.profiles.<name>: {kind: host_peer, family: claude,
  role: swe_reviewer, weight: supplementary, ...}` + a built-in profile (named to
  avoid colliding with a likely user profile).
- **Results:** tag host_peer findings same-family + supplementary in the v1.19.0
  results record.
