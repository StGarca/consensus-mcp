# v1.18.0 — Extensible, cross-platform contributor selection (design spec)

**Status:** seed design for the 4-way Workflow A consult (claude + codex + gemini + kimi).
**Branch:** v1.18.0. **Author:** claude (orchestrator). **Date:** 2026-05-22.

## 1. Motivation

Today the contributor set is hard-wired: `contributors/` ships `claude.py`,
`codex.py`, `gemini.py`; kimi is allow-listed (`config.py:147`) but has **no
adapter** (config.py:145 says "tracked separately") and is only dispatchable via
a monkeypatch wrapper that lives in a *different* project
(`ebook2audiobook/run/_dispatch_kimi.py`). Adding a 5th AI means writing a new
adapter module + a release. Operator goal: **at `consensus init`, let the user
pick which AIs to run (any number), and let the set grow later without code.**

## 2. Decisions already made (operator)

- **Config-driven profiles**, not per-AI Python adapters. Adding an AI = adding a
  profile (data), no new code. This **subsumes a bespoke `KimiAdapter`** — kimi
  becomes a built-in profile.
- **Detect + guide** for setup (no auto-exec): detect which selected CLIs are
  installed; for missing ones print OS-appropriate install + login commands.
- **Claude is fully optional** (strict "any AI, min 2" open-contributor model);
  panels with no claude are legal.
- **Cross-platform + any Python ≥3.11** (the old-Python constraint was the
  audiobook app's, not ours).
- Ships as **v1.18.0** (minor — feature). v1.17.5 already cut.

## 3. Requirements

R1. A **profile** fully describes how to detect, invoke, parse, install, and
    authenticate one AI — as data.
R2. A **generic adapter** consumes a profile and satisfies the
    `ContributorAdapter` interface, **reusing the existing dispatch machinery**
    (`_dispatch_base`, T6 seal) rather than reimplementing it.
R3. Built-in profiles ship for the 4 known AIs; users add more via config
    (`contributors.profiles` — `config.py:353` already reads a
    `contributors.adapters` map).
R4. `consensus init` offers **interactive multi-select** of available profiles,
    pre-checks detected CLIs, **enforces min-2**, writes `contributors.enabled`.
R5. A **non-interactive** path (`--contributors a,b,c`) preserves scripted/CI
    installs (the wizard already honors flags-over-prompts).
R6. **Detect + guide**: OS-aware; missing CLIs get exact install + auth commands
    printed; nothing is executed.
R7. Committed `.mcp.json` is **cross-platform** (drop Windows-only `py -3.11`);
    `requires-python` → `>=3.11`.
R8. No behavior change for existing installs that already pin
    `contributors.enabled` in config.

## 4. Profile schema (proposed)

```yaml
# built-in: consensus_mcp/contributor_profiles/<name>.yaml  (or a Python dict)
name: kimi
kind: cli_reviewer          # cli_reviewer | host  (claude = host, in-process)
detect:
  command: kimi             # binary resolved via shutil.which (OS-aware)
invoke:
  transport: stdin          # stdin | flag
  base_args: ["--quiet", "--thinking"]
  prompt_flag: null         # transport=flag → the flag (e.g. gemini "-p"); stdin → null
  workdir_flag: "-w"        # optional; sets cwd so the CLI loads project context
env:                        # injected into the subprocess
  # GEMINI_CLI_TRUST_WORKSPACE: "true"   # (gemini)
output:
  strip_patterns:           # chrome to remove before JSON parse
    - "\\n*To resume this session:\\s*kimi -r \\S+\\s*$"
install:                    # detect+guide ONLY — never auto-run
  windows: "pipx install kimi-cli"
  linux:   "pipx install kimi-cli"
auth:
  command: "kimi login"
  note: "OAuth/subscription login (interactive)"
timeout_seconds: 1800
```

`claude` is `kind: host` (no CLI, no install — it's the running Claude Code
environment; its "review" is the orchestrator's own analysis via the existing
`claude_artifact_callback`). codex/gemini/kimi are `cli_reviewer`.

> Install commands above are **illustrative** — each built-in profile's
> install/auth strings MUST be verified against the real CLI before shipping
> (verify-before-invent); the consult/impl confirms exact package names.

## 5. Architecture & the ONE open question for the consult

The existing `_dispatch_codex.py` / `_dispatch_gemini.py` are heavyweight:
streaming, stall-detection, retries, schema validation, T6 sealing. A generic
profile adapter must **not** throw that away. Three integration options — **this
is the central question for the consult:**

- **(A) Extract-and-parameterize.** Pull the shared core into `_dispatch_base`
  (already partly there), make codex/gemini/kimi/new-AIs all thin
  profile-driven invokers over it. Cleanest end state; highest risk to
  battle-tested code; biggest diff.
- **(B) Additive generic adapter.** Leave codex/gemini modules as-is; add a
  `ProfileAdapter` that handles the (simpler) stdin/flag invoke + parse for kimi
  and any new AI, reusing `_dispatch_base` for prompt-build + seal. Lowest risk;
  some duplication; codex/gemini keep bespoke modules.
- **(C) Profile-as-front, modules-as-back.** Profiles are the single source of
  truth for *all* AIs; codex/gemini profiles declare `module: _dispatch_codex`
  so the generic adapter delegates to the specialized module, while pure-data
  profiles (kimi, new AIs) use the built-in invoker. Uniform config surface,
  moderate complexity.

**Author's lean: (B) for v1.18.0** (ship the extensibility + kimi-as-profile at
low risk), with (A) as a tracked follow-up if duplication bites. The consult
should pressure-test this, plus: profile schema completeness (does it capture
codex token-auth, gemini trust-workspace, kimi stdin-not-`-p`?), where profiles
live, and provenance correctness (the parent kimi wrapper mislabels model as
`gemini-2.5-pro` — a real `ProfileAdapter` must seal **accurate** provenance).

### Registry integration
`_engine_factory`: `build_adapters(config)` reads `contributors.enabled` +
merges built-in profiles with `contributors.profiles` from config, and
instantiates `ProfileAdapter(profile)` (or delegates per option C). Min-2 is
validated in `config.validate` (already enforces the open-contributor floor).

## 6. `consensus init` selection UX (`_init_wizard.py`)

1. Build the available-profile list (built-ins + config profiles).
2. For each, detect the CLI (`shutil.which`, OS-aware) → show `✓ installed` /
   `✗ missing`.
3. Interactive multi-select (default: pre-check detected). Re-prompt until **≥2**
   selected. `claude` is just another option (optional).
4. Write `contributors.enabled` to `.consensus/config.yaml`.
5. Run **detect + guide** for the selection: for each missing CLI print
   `install[os]` + `auth.command` + note. Nothing executed.
6. Non-interactive: `--contributors a,b,c` skips the prompt (flags-over-prompts,
   already the wizard's contract); still prints detect+guide.

## 7. Cross-platform install (R7)

- Committed repo-root `.mcp.json`: `py -3.11 -m consensus_mcp.server` →
  `consensus-mcp` entry point (on PATH both OSes). `consensus-init`'s
  `_resolve_mcp_command` is already OS-correct (entry-point → `sys.executable`
  fallback); no change needed there.
- `pyproject.toml`: `requires-python = ">=3.10"` → `">=3.11"`; reconcile the
  README "Python 3.10–3.12" line.

## 8. Testing

- Profile schema validation (required fields, transport enum, OS keys).
- `ProfileAdapter._invoke` against a **fake CLI** (stdin + flag transports),
  using the existing fake-subprocess idiom; assert env injection + output
  strip + JSON parse + T6 seal with **correct provenance**.
- Wizard: min-2 enforcement, flag-precedence (non-interactive), config write.
- detect+guide: monkeypatch `sys.platform` + `shutil.which`; assert the right
  per-OS install/auth lines for missing CLIs and silence for present ones.
- Cross-platform `.mcp.json` generation; `requires-python` floor.
- Regression: existing `contributors.enabled` configs build unchanged (R8).

## 9. Non-goals

- Auto-executing installs or auth (operator chose detect+guide).
- A GUI / TUI selector (plain CLI multi-select).
- Rewriting codex/gemini dispatchers (unless the consult picks option A).
- Fixing the parent project's kimi wrapper (we replace its need with a profile).

## 10. Process

Spec → self-review → operator review → **4-way Workflow A consult**
(propose-converge; the open question in §5 is the crux) → converged-plan →
TDD implementation → Workflow B audit → cut as part of v1.18.0.

## 11. Consult outcome (converged 2026-05-22)

4-way Workflow A consult (claude + codex + gemini + kimi; sealed proposals in
`consensus-state/.../iteration-v1180-contributor-design-2026-05-22/`). Integration
vote split **2-2 (B: codex, kimi · C: gemini, claude)** — resolved by
**weighted-synthesis, not vote-count**, because the labels overstated the gap:
all four proposals author YAML profiles for **all four** AIs, so C's uniform-
surface goal is met under B too. The only real difference was where built-in
dispatch routing lives — and kimi's "a `module:` pointer leaks implementation
detail into config" + codex's "two execution modes behind one adapter surface"
make **B-routing the cleaner choice**.

**Converged decision — B-ROUTING + UNIVERSAL PROFILES:**
1. Author built-in YAML profiles for all four AIs (wizard list, detect status,
   install/auth guidance, model/provenance, forward-compat for a future option A).
2. Dispatch routing stays in `_engine_factory`: `_BUILTIN_ADAPTERS`
   (claude/codex/gemini) keep their existing classes **unchanged**; a new generic
   `ProfileAdapter` serves `kind: cli_reviewer` profiles that aren't built-in
   classes (kimi + user-added AIs). **No `module:` field.** Option A is a tracked
   follow-up.

Unanimous: schema additions (`model`, `auth.*`, `sealed_filename`, `id_prefix`,
`output.schema_enforced`); profile-driven provenance (fixes the kimi
`gemini-2.5-pro` mislabel); packaged built-ins + `contributors.profiles` overlay;
numbered multi-select wizard (min-2, claude-optional, `--contributors` flag);
OS-aware detect+guide (no exec); R7 cross-platform; and **verify-before-ship** on
all install/auth strings (no invented package-manager commands).

**Added post-consult — per-AI instruction files:** each selected AI is seeded
with shared operating guidelines (vendored from the Karpathy
`andrej-karpathy-skills/CLAUDE.md`, verified HTTP 200) in its convention file —
**claude→`CLAUDE.md`, codex→`AGENTS.md`, gemini→`GEMINI.md`, kimi→`AGENTS.md`**
(codex+kimi share `AGENTS.md` → write once). Content vendored into the repo
(`consensus_mcp/contributor_instructions/base.md`, source cited), not fetched at
init. Provisioning behavior (non-destructive managed-block vs separate file vs
overwrite) is the one open operator decision. See the full
`converged-plan.yaml` in the iteration dir.
