# Repo Audit and Improvement Plan: consensus-mcp v2.2.1

Audit of HEAD ebd13dc (main, = v2.2.1 tag + one README doc commit), 2026-07-01.
Method: fable_review.md rubric, executed as an 8-dimension multi-agent audit
(architecture, code quality, security, testing, performance, dependencies,
devex/ops, docs) with every finding independently adversarially verified
against the tree (evidence, consequence, and severity calibration re-checked;

Read-only audit: no code was modified.

---

## 1. Executive Summary

Overall health grade: B. The fundamentals are unusually strong for a
single-maintainer tool: a clean adapter architecture, a hardened security
posture (argv-only subprocess, default-deny env scrubbing, one blessed atomic
writer, live re-validated seals), 2,257 hermetic tests on a 4-leg CI matrix,
and provenance discipline that most teams never reach. What keeps it from an
A is that the tool's own trust surface is under-protected: the parallel
reviewer fan-out can silently lose seal-index and audit-log records to a
lost-update race, five live MCP tool modules (~1,850 LOC including the
production gate and ledger writer) have zero CI-run tests, and the
most fix-prone code in the repo (the dispatcher supervision loop) exists as
four near-verbatim copies that must each receive every future fix.

Top 3 risks: (1) the seal-index/audit-log concurrent-write race - dropped
provenance records in a tool whose pitch is sealed provenance; (2) untested
trust surface - regressions in gate/ledger/section-write tools and the MCP
protocol envelope merge green on every commit; (3) quadruplicated
supervision/parser code - the historically most-patched logic diverges one
dispatcher at a time.

Top 3 opportunities: (1) port the 60-probe smoke suite to pytest so the whole
tool surface rides the existing 2-OS CI for free; (2) unify the ~15 repo-root
resolvers and 4 supervision loops onto single root primitives - the project's
own "fix the class, not the site" doctrine applied to its two biggest
remaining classes; (3) a docs-truth pass plus a dev extra - the README
currently teaches a command that errors and omits a supported reviewer, which
is a first-session credibility cost for an auditability product.

## 2. Repo Map (Phase 1)

Purpose: consensus-mcp puts multiple AIs on a codebase in two modes.
Consult: a panel of external AI CLIs (codex, gemini/agy, grok, kimi)
independently reviews changes; convergence gates + sealed YAML artifacts +
delivery tokens make the outcome auditable. Build (v2.x architect loop): a
top-tier model plans/oversees while a cheaper model implements in a contained
git worktree. Distributed via pipx/pip as an MCP server + 14 console scripts.
Maturity: released (v2.2.1), dogfooded heavily (134 sealed iterations in
consensus-state/), single maintainer.

Stack: Python 3.11+, setuptools, runtime deps only pyyaml + jsonschema.
Hand-rolled stdio JSON-RPC MCP server (deliberate, documented at
server.py:19-23). CI: GitHub Actions test.yml, pytest on ubuntu+windows x
py3.11/3.12. No lint/format/type-check tooling configured anywhere.

Architecture sketch (unidirectional, verified by import scan + live import):

    server.py (hand-rolled JSON-RPC, ~28 tools via tool_registry)
      -> tools/            thin MCP wrappers
      -> workflow_engine   phase fan-out (ThreadPoolExecutor since v1.40)
      -> contributors/     adapter per AI (registry in _engine_factory)
      -> _dispatch_*.py    external-CLI supervision (the 4 big files)

    validators/, _snapshot_state, _atomic_io: trust primitives
    looper_plan/ (vendored, zero inward imports), claude_extensions/ (skills
    + hooks installed into ~/.claude)

## 3. Audit Report (Phase 2)

Verification legend: every finding below survived an independent adversarial
verify pass. Severities shown are post-verification. [fact] = verifiable in
the file; [judgment] = assessed opinion.

### The ugly parts (utmost priority)

These four High findings are where real damage can occur:

H1. Seal-index/audit-log lost-update race under parallel fan-out
[performance/correctness, fact, CONFIRMED]
- Where: workflow_engine.py:84-92, :264-270, :331-337; _dispatch_base.py:816-823
- What: since v1.40.0, _dispatch_phase_parallel runs each contributor's full
  dispatch in a ThreadPoolExecutor worker; every dispatcher ends with
  _seal_via_t6 -> review_write_and_seal, which does a read-modify-os.replace
  of the shared archive index.yaml (321KB in this repo) and appends audit
  events. Two reviewers finishing inside the same read-to-replace window both
  load the same index version; the second replace silently erases the first
  reviewer's entry. Verified mechanically at every cited line.
- Why it matters: silent provenance loss in the product's core promise. The
  atomic writer prevents torn files but not lost updates; nothing serializes
  concurrent sealers across threads/processes.

H2. Repo-root resolution is fragmented across ~15 divergent resolvers; the
divergence concretely breaks apply_codex_patch under the standard pipx
bootstrap [architecture, fact, CONFIRMED]
- Where: _paths.py:39-62 (the intended single source of truth), server.py:58,
  _dispatch_base.py:163, _self_drive.py:125-129, tools/apply_codex_patch.py:86,
  plus ~10 more (full list verified by grep; includes 3 hook variants).
- What: 15 _resolve_repo_root/_repo_root definitions coexist with different
  env-var precedence and fallback semantics. The apply_codex_patch failure was
  reproduced empirically (fails closed under pipx bootstrap). _self_drive
  still ships the exact Path(__file__).parent.parent fallback that
  _dispatch_base's own comments document as unsafe (resolves to site-packages
  in a pipx install).
- Why: the same cwd+env resolves different roots depending on entry point, so
  containment gates, archive writes, and config discovery can silently
  disagree within one pipeline invocation.

H3. Streaming supervision loop and output parsers are copy-pasted across the
four dispatchers [code quality, fact, CONFIRMED with reproduced measurements]
- Where: _dispatch_gemini.py:330-508, _dispatch_grok.py:472-631,
  _dispatch_kimi.py:946-1097, _dispatch_codex.py:411-571
- What: the Popen reader threads, stdin writer, abort polling, silence
  watchdog, heartbeat, and wall-time ceiling are near-verbatim copies
  (stdout_reader: 4 copies at 0.96 avg pairwise similarity; parsers 0.82-0.87).
  The replicated fix comments show deadlock/watchdog fixes were applied 3-4
  separate times each.
- Why: this is the historically most fix-prone code in the repo; every future
  bug must be found and fixed four times or it silently persists.

H4. Five live MCP tool modules (~1,857 LOC) have zero pytest coverage; their
only checks live in _smoke_test.py, which CI never runs [testing, fact,
CONFIRMED]
- Where: tools/repo_set_section.py (520 LOC),
  tools/gate_evaluate_production_with_scope_match.py (495),
  tools/state_update_decision_ledger.py (403), tools/_md_sections.py (251),
  tools/repo_get_section.py (188); registered live at server.py:151-167.
- Why: a regression in the production-scope gate, ledger writer, or spec
  section atomic write merges green on both CI platforms; the only
  compensating check is a manual single-platform release-time script.

### Architecture and design

- (Medium, fact, ADJUSTED) Foreign projects tracked at repo root: DGX
  work-queue system (dgx1_coordinator.py:1-14, dgx2_worker.py:5-8 - imports
  requests, declared nowhere; defaults to http://192.168.50.1:9000),
  deploy/ systemd units whose ExecStart hard-references
  %h/src/consensus-mcp (deploy/dgx1-coordinator.service:8-9), plus the Siege
  Engine game (index.html, siege-engine-gdd.md). All landed via commits
  90988bf/678b7a6 and are in the public v2.2.1 release tag. Nothing in the
  package references them. Consequence: internal LAN topology published;
  release diffs mix foreign changes; cleaning the repo can break the live
  systemd service that points into this checkout.
- (Low, fact, ADJUSTED) Layering inversion held together by lazy imports: the
  dispatch core reaches up into the tools layer for sealing
  (_dispatch_base.py:816 imports tools.review_write_and_seal inside
  _seal_via_t6; same in contributors/claude.py:95,
  contributors/host_peer_adapter.py:169). The sealing service is core domain
  capability living in the MCP-surface layer; import order is silently
  load-bearing.
- (Medium, judgment, CONFIRMED) Internal integration is in-process CLI
  emulation: adapters build argv, capture stdout via a permanently-installed
  thread-local stdout proxy (contributors/base.py:43-74), and JSON-decode
  whatever was printed. The de facto internal contract is "exactly one JSON
  object on stdout"; any stray print in ~5.7k LOC of dispatcher code corrupts
  a dispatch into a parse error.
- (Medium, judgment, CONFIRMED) God modules where the project historically
  breaks: _init_wizard.py (2,855 LOC, 76 defs, 9+ separable responsibilities),
  _smoke_test.py (3,339), _dispatch_kimi.py (1,837).
- (Low, judgment, CONFIRMED) Hand-rolled strictly serial stdio JSON-RPC
  server pinned to protocolVersion 2024-11-05 (server.py:286): one
  multi-minute tool call blocks the whole MCP session (no cancellation,
  progress, or timeouts). Acceptable today (dogfoods via shell binaries);
  it is the ceiling on MCP-native UX.

### Security (healthy - three modest findings)

The posture is strong: argv-list subprocess everywhere in dispatchers (no
shell=True), default-deny env allowlist with a hard-floor credential scrub,
yaml.safe_load exclusively, no pickle/eval/exec, one blessed atomic writer
(O_CREAT|O_EXCL + fsync + os.replace) reused by every marker/seal/config
writer, live re-validated design-approval seals, path-traversal guards with
resolve()+relative_to. Findings:

- (Low, fact, ADJUSTED) The PreToolUse protected-install guard
  (_is_protected_install_path, the one check documented as ALWAYS-ON) is only
  invoked for Edit tools (consensus_pretooluse_gate.py:487-494); Bash
  file-writes to ~/.claude/settings.json or the hook scripts are not checked,
  and in the default dormant state Bash is not gated at all.
- (Low, judgment, CONFIRMED) server.py:314 dispatches handler(**arguments)
  straight from JSON-RPC params; declared input_schema (additionalProperties,
  required, types) is never enforced - handlers that trust their schema are
  directly reachable with wrong-typed args.
- (Low, judgment, CONFIRMED) shell=True on operator-authored verification
  commands from .consensus/config.yaml (architect_loop_step.py:630-642) and
  goal_packet gates (_self_drive.py:374-384) - consistent with the documented
  trust model, but the config file supplying the command is gate-exempt.

## 4. Improvement Strategy (Phase 3)

Theme 1: One truth, many copies.
Most High findings are one theme: load-bearing primitives duplicated per
site - 15 repo-root resolvers, 4 supervision loops, 4 wrapper schemas, UTF-8
hardening in 1 of 13 entry points. The project already codified the
principle ("fix the class at one root primitive, not per-site") for security
classes; apply it to these two remaining classes.
Target state: exactly one resolver (_paths.py, parameterized), one supervised
subprocess runner in _dispatch_base, one entry-point bootstrap helper.
Done when: grep finds 1 definition of each; a parametrized test asserts all
entry points resolve the same root for the same cwd/env.

Theme 2: The trust surface must ride CI.
The product's promise (gates, seals, ledger) is exactly where automated
coverage is thinnest: 5 untested live tools, untested protocol envelope,
manual-only smoke + release gates, a manifest guard with a known blind spot.
Target state: everything a release depends on runs on every push, both OSes.
Done when: the 5 modules + _handle_request have pytest coverage; smoke checks
are pytest-marked integration tests in CI; the release gate is a thin wrapper
over CI-green + packaging checks; py3.13 leg green.

Theme 3: Concurrency outran the storage layer.
v1.40.0 parallelized dispatch but left single-writer read-modify-replace
files underneath (index.yaml, audit log), skipped advisory mode, and logs
grow unbounded.
Target state: a single cross-process lock primitive around every
read-modify-replace of shared state; all three engine modes parallel;
retention policy on append-only logs.
Done when: a two-writer concurrency test passes 1,000 iterations; advisory
wall-clock ~= slowest reviewer; dispatch-log bounded.

Theme 4: Docs are part of the release, and drift is a shipped bug.
Six verified cases of docs teaching wrong commands/surfaces. For a tool that
sells auditability, doc truth is product truth.
Target state: user-facing claims that are cheap to check are checked by
tests (command exists in [project.scripts], flags exist in argparse);
one env-var reference; RELEASING.md.
Done when: a test walks README/CHANGELOG fenced commands against the script
table; packaged docs list the real tool surface.

Explicitly NOT fixing (trade-offs):
- Hand-rolled MCP server (serial, pinned protocol): documented decision,
  correct for current dogfooding; revisit only when MCP-native cancellation
  or an SDK adoption is wanted. Cost of a rewrite now exceeds payoff.
- shell=True on operator-authored verification commands: consistent with the
  trust model; document it, do not redesign.
- In-process CLI emulation as the internal API: real weakness, but a typed
  internal API is an XL refactor across all adapters; defer until it next
  causes a concrete failure. The thread-local stdout proxy works.
- Kimi per-dispatch clone/snapshot cost: design is sound and budgeted;
  already queued as the engine-shared-snapshot design item - consult first
  (staleness vs concurrent edits), do not micro-optimize ahead of it.
- Full-strict mypy: adopt incrementally on the 85% base; do not gate
  releases on it this cycle.
- consensus-state/ tracked in git (50MB, 134 iterations): deliberate
  auditability choice; revisit only if clone size becomes a complaint.

### Implementation sketches (top 3)

M1.1 Seal-index lost-update fix.
Approach: a tiny cross-process mutex in the atomic-IO layer - lock directory
via os.mkdir (atomic on POSIX+Windows, no fcntl/msvcrt divergence) next to the
target file, bounded retry with jitter, stale-lock takeover after a timeout
(holder writes pid+timestamp inside). Wrap the full read-modify-os.replace of
index.yaml and the audit append. Key steps: (1) land M0.3 reproducer;
(2) add acquire/release + stale takeover in _atomic_io; (3) use it in
review_write_and_seal + audit_append_event; (4) hammer test both OSes.
Gotchas: hold window must stay tiny (no dispatch work inside the lock);
in-process ThreadPoolExecutor workers AND separate CLI processes both write,
so a threading.Lock is insufficient - it must be a filesystem lock; beware
WSL/NTFS mkdir latency in the retry tuning. Consult fork: mkdir-lock vs
O_EXCL lockfile vs single-writer queue.

M1.2 Repo-root unification.
Approach: make _paths.resolve_repo_root(env_vars=..., require_marker=...,
allow_cwd_walk=...) the only implementation; each entry point states its
precedence declaratively. Migrate the two proven-broken sites first
(apply_codex_patch pipx failure; _self_drive site-packages fallback), then
mechanical migration of the remaining ~13 with PEP 562 shims for
compatibility. Add one parametrized test: for a matrix of (cwd, env) cases,
every entry point resolves the identical root. Gotchas: the three hook
variants deploy detached into ~/.claude and CANNOT import the package - they
need a vendored copy stamped by the installer (add a sync test so the
vendored copy cannot drift); _dispatch_base's fail-closed semantics differ
deliberately from server.py's fail-open - the consult should bless one
precedence table rather than preserving all 15 behaviors.

M2.1 Supervision-loop extraction.
Approach: diff all four copies FIRST to build the superset feature matrix
(grok self-cancel recovery, kimi tmpfs, codex pre-first-byte kill, gemini
agy quirks) - the four copies embed 3-4 rounds of divergent historical fixes,
and the extraction must be the union, not the intersection. Define
run_supervised(cmd, stdin_payload, on_line, watchdog_cfg, ceiling_cfg,
terminate_tree) in _dispatch_base; port gemini first (smallest delta), run
its full test file, then grok, kimi, codex one PR each. Keep parsers
per-dispatcher. Gotchas: watchdog timing tests are the flakiest surface -
keep the existing per-dispatcher tests as the port's acceptance gate;
do not change event names in dispatch-log.jsonl (readers depend on them).
